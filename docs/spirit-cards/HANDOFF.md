# 霊宝コレクション（仮）— 開発引き継ぎ

> HUNTER×HUNTERの「グリードアイランド」に着想を得た、スマホ向けカード収集 × プレイヤー間交換ゲーム。
> HxHのIPは使わず、オリジナルの「失われた霊宝を集める」世界観で作る。

---

## 1. このプロジェクトは何か

- **プラットフォーム**：スマホアプリ
- **ジャンル**：限定供給型のコレクションカードゲーム（＋プレイヤー主導の交換経済）
- **再現したい中心体験**
  - カード収集のワクワク感
  - プレイヤー同士の交換と駆け引き

## 2. ゲームの核（最重要・ここを外すと別物になる）

**カードの「世界総数」を有限にする。** これが設計の心臓部。

- 普通のガチャ ＝ 同じカードが無限に出る
- このゲーム ＝「この秘宝は世界に数枚しか存在しない」
- だから自力では絶対に揃わず、他プレイヤーと交換・交渉する必然性が生まれる

→ 「収集のワクワク」と「交換の駆け引き」が **同時に** 成立する仕組み。
グリードアイランドが面白かった理由そのもの。

## 3. 現状（試作の到達点）

`spirit_cards_prototype.jsx` に、単一画面のReact試作がある。コアループを実際に触って確認できる。

- **3タブ構成**：台帳（コレクション一覧）/ 探索（カード入手）/ 取引所（交換）
- **カード16種**、レアリティ3段階（普通8・希少5・秘宝3）
- `CARDS` 配列の `wild` フィールド ＝「野生（探索）に存在する総数」
  - 秘宝は `wild: 0`。探索では入手不可 → **取引必須**になるよう設計してある
- 探索は `weightedPick()` による重み付き抽選。在庫（`wild`）が尽きると出なくなる
- 取引はNPC（ボット）との1対1単純交換 ＋「粘る（交渉）」

### 試作で割り切っている点（＝本番で解決すべき宿題）

- 交換相手がNPC → 本物の希少性・駆け引きが薄い。**実プレイヤー間通信が必要**
- 1対1の単純交換のみ（複数カード提示・オークションなし）
- 呪文カード（収納／実体化／同行／帰還 など）未実装
- クエスト・条件付き入手なし
- データはメモリ上のみ（保存・同期なし）

### カード構成（参考）

| レアリティ | 枚数 | 探索で入手 | 例 |
|---|---|---|---|
| 普通 | 8 | ○（在庫多め） | 灯火の精・川の囁き 等 |
| 希少 | 5 | ○（在庫少） | 月光の鏡・霧の女王 等 |
| 秘宝 | 3 | ✕（取引のみ） | 不死鳥の心臓・創世の宝珠 等 |

## 4. 本番アプリへの技術メモ（未決 ＝ 着手前に決める）

最大の論点は **「リアルタイムなプレイヤー間取引をどう実現するか」**。

おすすめの出発点（あくまで叩き台）：

- **フロント**：Expo（React Native）でiOS/Android両対応。試作のReactロジックを移植しやすい。
  「まずWebで」ならPWAで割り切る案もアリ
- **バックエンド**：最初は Supabase または Firebase で爆速に立ち上げる（認証・DB・リアルタイムが揃う）
- **DB設計の肝**：「カードの世界在庫」をサーバーで一元管理する。
  誰が何枚持っているかの総和＝世界総数が固定 ── これがGI的希少性の生命線
- **取引処理**：1トランザクションで2人の所持を原子的に入れ替える。二重取得・消失を防ぐ

想定テーブル（最小）：
- `cards`（マスタ：id, name, rarity, world_supply）
- `user_cards`（誰が何を何枚持つか）
- `trades`（取引の提案・成立ログ）

## 5. 次の一手（おすすめ順）

1. **ゲームデザイン仕様を1枚に固める**（カード総数・レアリティ配分・入手手段・勝利条件）
2. **データモデルを確定**（上記テーブル）
3. 試作のロジック（探索の重み付き抽選・在庫管理）を**サーバー側に移す設計**に書き換え
4. まずは「実在庫を持つ単一プレイヤー＋複数ボット」をサーバー接続で再現 → その後リアル多人数へ
