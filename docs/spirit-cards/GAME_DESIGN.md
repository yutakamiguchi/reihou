# 霊宝コレクション（仮）— ゲームデザイン仕様 v0.1

> 本書は HANDOFF.md「5. 次の一手 ①」に対応する、設計を1枚に固めるための仕様書。
> **目的：実装に入る前に「有限供給という核」を数理的に破綻しない形で言語化する。**
> 数値は MVP 想定（同時運用プレイヤー 50〜200 人規模）の初期値。`§9 要決定事項` が貴方の判断待ち。

---

## 0. 一行サマリ

「世界に存在するカード総数を固定し、特に秘宝(legend)は探索で一切増えない。
だから自力では台帳を埋められず、プレイヤー間の交換・交渉が必然になる」
── これが唯一にして絶対の核。以下のすべてはこの核を守るための具体化。

---

## 1. 用語の定義

| 用語 | 意味 |
|---|---|
| **カード種 (card type)** | 「灯火の精」など、図鑑の1マスに当たる種類。MVPで **16種**。 |
| **個体 (unit)** | カード種の実体1枚。所有・取引の最小単位。 |
| **世界総数 `world_supply`** | あるカード種が、その季節の世界に存在する個体の総数。**固定値**。 |
| **世界在庫 `world_reserve`** | まだ誰にも所有されていない（探索で掘り出される前の）個体数。 |
| **所有合計 `owned_total`** | 全プレイヤーが持つそのカード種の合計枚数。 |
| **季節 (season)** | 1つの「世界」。季節ごとに `world_supply` を再設定して新しい世界を開く（§7）。 |

> **基本恒等式（最重要）**
> `world_supply(c) == world_reserve(c) + owned_total(c)`  ── どの瞬間も常に成立する。
> 探索＝在庫から所有への移動。取引＝所有間の移動。**カードは増えも減りもしない。**

試作の `wild` は本書の `world_reserve` 初期値、`foundWild[id]` は「掘り出した個体数」に相当する。

**世界に存在するカード総数（季節「原初」）= 普通2,150 + 希少170 + 秘宝9 = 2,329 個体。**

### 設計意図

- **普通(8種)＝基軸通貨/素材**：在庫を厚くし、誰でも探索で手に入る。交換の「お金」役。
- **希少(5種)＝中盤の追跡対象**：在庫は薄い。序盤は探索で出るが早期に枯渇 → 以後は交換でしか動かない。
- **秘宝(3種)＝終盤・名誉**：`world_reserve=0` なので**探索では永遠に出ない**。世界に各3個体しか存在しない。
  - これが「自力で台帳を埋められない」を**構造的に保証する**部分。

> **核となる数理的帰結**：秘宝は各3個体。台帳コンプには3種すべての秘宝が1枚ずつ要る。
> 秘宝1種につき3枚しかない以上、**全16種コンプを同時に達成できるプレイヤーは世界に最大3人**。
> （3人が各々「不死鳥1・星詠み1・創世1」を握れば3組成立、それが上限）
> この「席は3つしかない」緊張が交換経済の駆動力になる。

## 3. 入手経路マトリクス

| 経路 | 普通 | 希少 | 秘宝 | 供給を増やすか |
|---|---|---|---|---|
| **探索（ソロ）** | ○（在庫ある限り） | ○（在庫ある限り） | ✕ | 増やさない（在庫からの移動） |
| **プレイヤー間取引** | ○ | ○ | ○ | 増やさない（所有間移動） |
| **秘宝の世界投入（季節開始時のみ）** | ✕ | ✕ | ○（faucet） | 季節開始時に1回だけ。§9-A で方式決定 |
| クエスト報酬（post-MVP） | ○ | △ | ✕ | 在庫から出すので増やさない |

**供給を生む蛇口(faucet)は実質「季節開始時の在庫設定」だけ。** 運用中にカードが純増する経路を一切作らないことが核の維持条件。

## 4. 探索（ソロ・カード入手）

試作 `explore()` をサーバー権威化した版。

- スタミナ（霊力）を1消費して1回抽選。
- 抽選は **在庫が残るカード種のみ** を対象に、`重み = レアリティ係数 × world_reserve(c)` で重み付き抽選。
  - レアリティ係数：普通=10, 希少=3, 秘宝=0（係数0なので秘宝は理論上も絶対に当たらない＝二重の安全装置）。
- 当選したら原子的に `world_reserve(c) -= 1; owned[player][c] += 1`。在庫が0なら対象から外れる。
- 全対象の在庫が尽きたら「この地に野生の霊宝は残っていない → 取引所へ」。

### スタミナ設計（試作の穴を塞ぐ）

> 試作は「気を練る」で**無制限に全回復**でき、探索回数に制約が効いていなかった。
> 在庫上限はあるが、回数無制限だと早い者勝ちで在庫を一瞬で枯らせてしまう。

- スタミナ上限 24、**時間回復**（例：10分で+1、フル回復に4時間）。手動全回復は廃止。
- これにより「探索の総量＝時間」で律速し、在庫の枯渇速度と希少性をコントロールする。
- 課金/広告での回復は将来検討（§9 外）。MVPは時間回復のみ。

## 5. 取引（プレイヤー間交換）

### MVP

- **1対1・複数枚提示可の相互合意トレード**（試作の1枚:1枚から拡張）。
  - 提案者が「渡すカード群」と「求めるカード群」を提示 → 相手が承認/拒否/カウンター提案。
- **成立処理は単一トランザクションで原子的に所有を入れ替える**（§6 INV-4）。
- 提案中のカードは**エスクロー（出品ロック）**し、二重提案・二重消費を防ぐ。
- NPCボット：MVPでは「実在庫を持つボット」を数体置き、過疎時の流動性を確保（HANDOFF §5-④）。
  - ボットも `owned` を持つ正規プレイヤー扱い。ボットが秘宝を無から作ることはしない。

### post-MVP（設計予約のみ）

- オークション/入札、掲示板型の売り注文、交渉チャット、取引手数料による sink。

### 交渉について

試作の「粘る（haggle）」はNPC向けの乱数演出。実PvPでは**カウンター提案の往復**が交渉そのものになるため、乱数haggleは廃し、提案↔カウンターのUIに置き換える。

## 6. 経済の不変条件（Invariants）── テストで常時検証する

実装の全トランザクションは以下を破ってはならない。CIの不変条件テストとして固定化する。

- **INV-1 保存則**：任意のカード種 `c` について `world_reserve(c) + Σ_user owned(user, c) == world_supply(c)`。**常に**。
- **INV-2 非負**：`world_reserve(c) >= 0` かつ全 `owned(user, c) >= 0`。
- **INV-3 探索の健全性**：探索でカードを得るのは `world_reserve(c) > 0` のときのみ。得たら reserve を1減・owned を1増（同一Tx）。
- **INV-4 取引の原子性**：取引はA・B双方の owned を1つのTxで増減。途中失敗時は全ロールバック。reserve は不変。
- **INV-5 秘宝の封印**：`rarity == legend` のカードは探索経路で reserve から減らない（係数0かつ reserve=0 の二重保証）。
- **INV-6 エスクロー整合**：出品ロック中の枚数は他取引・探索消費に回らない。`owned = 自由枚数 + ロック枚数`。

> これらが守られる限り「世界総数は固定」が機械的に保証される。逆に言えば、新機能はこの6条件を破らないかだけ確認すればよい。

## 7. 季節（シーズン）制と勝利条件

固定供給は「いつか誰かが秘宝を死蔵して場が固まる」リスクを持つ。**季節制で世界をリセット**して回す。

- 1季節 = 一定期間（例：4〜8週間）。季節ごとに新しい `world_supply` で新世界を開く。
- **勝利/到達目標を多層化**（全員が秘宝3枚を狙う設計だと大多数が報われないため）：
  - **収集家（誰でも到達可）**：普通8＋希少5＝**13種コンプ**。基本の達成感ライン。
  - **蒐集者（最大3名/季節）**：**全16種コンプ**。称号と季節殿堂入り。
  - 季節終了時にスナップショットを取り、ランキング・称号を確定 → 次季節へ。
- 季節をまたいだカードの扱い（持ち越し/記念化/失効）は §9-C で決定。

## 8. 呪文カード（GIの「呪文」相当）── MVP範囲外・設計予約

HANDOFF記載の 収納/実体化/同行/帰還 等。**MVPでは収集対象16種に含めない（world_supplyの対象外）**。
役割だけ予約しておく：

| 呪文 | 想定効果 | 優先度 |
|---|---|---|
| 収納/実体化 | カードの保管・場出し（UI上の概念） | 後 |
| 同行 | 他プレイヤーと一時的に探索を共有 | 後 |
| 帰還 | 安全地帯へ戻る/取引所即時アクセス | 後 |
| 強奪/盗難系 | カードを奪う（PvP緊張） | 要検討（経済影響大） |

> 強奪系は「奪っても総数は不変（所有移動）」なら INV を破らないが、体験設計の影響が大きいので別途。

## 9. 要決定事項（貴方の判断待ち）

実装前に決めたい論点。**A〜D が次の打ち合わせの議題**。括弧内は推奨デフォルト。

- **A. 秘宝の世界投入方法**（推奨：季節開始時に各3個体を「頂点クエスト報酬」または「ランダムな少数プレイヤーへ初期配布」）
  - 探索で出ない秘宝を、誰の手に最初に握らせるか。発見の物語性に直結。
- **B. 供給数のスケール方針**（推奨：普通=固定厚め、希少=固定薄め、**秘宝=人数によらず常に3**で希少性を維持）
  - 多人数化したとき普通/希少を人数連動で増やすか、固定のままスケール圧を希少性に転化するか。
- **C. 季節をまたぐカードの扱い**（推奨：旧世界のカードは「記念カード（収集対象外・図鑑に痕跡）」化、新世界はクリーン）
- **D. 技術スタック**（HANDOFF §4。フロント=Expo/PWA、バックエンド=Supabase/Firebase/自前Colyseus）
  - 本リポは Colyseus 資産あり。リアルタイム性とトレード原子性のどちらを重視するかで選ぶ。
